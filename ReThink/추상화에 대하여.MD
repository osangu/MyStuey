추상 팩토리 패턴을 공부하는 과정에서, 다시 한 번 추상화에 대해 다시 한 번 생각을 정리하였고 여기에 기록합니다. :airplane:  
어디까지나 개인적인 생각이므로, 불편한 부분이 있으시면 말씀해주세요.

----
**`추상화`**  
공통되는 부분을, 압축 또는 획일화 하는 것이라고 느낍니다.  

추상화를 왜 하는 걸까?  
1. 공통되는 부분을 압축하였으니 그렇지 않았을 때와는 비교적 관리가 쉬울 것입니다.
   + 클린코드에서 프로그래밍의 80%는 소위 유지보수라고 하는 구절을 기억해 보면, 이것만으로도 매우 큰 장점이라고 생각합니다
2. 캡슐화를 위해서
   + 이 부분은 밑에 캡슐화에서 보다 자세히 다루겠습니다.
----
**`캡슐화`**  
추상화를 하다 보면 부가적으로 따라오는 부분이라고 생각합니다.  

그럼 캡슐화란 무엇인가?

+ 추상화 단계가 높은 로직의 내부를 타인이 모르게 숨기는 것
  + 여기서 타인은 같은 내 코드를 사용하는 아무나 될 수 있습니다.
+ 추상화에 속할 만큼 공통되는 부분 또는 메인 로직이 아니지만, 저들을 위해 존재하는 부가적인 경우를 숨기는 것
  
보통 다른 이가 몰라도 되는 부분의 로직을 숨기는 것이라고 도출할 수 있겠습니다.

그럼 왜 사용하는가? 
+ 우리가 Xbox 또는 플스를 통해 게임을 할 때, 
+ 게임기 버튼과 조이스틱의 동작 원리를 알지 못해도 사용할 수 있는 것과 같이,
+ 내 코드의 동작 원리를 알지 못해도 기존의 사용법을 통해서 사용할 수 있게 하는 것입니다
  + 사용법이라 해봐야, 호출방법과 반환값 정도가 있을 것 입니다.

만약 게임할 때, 저 과정을 알아야 하면 게임할 사람이 그렇게 많지 않다고 생각합니다 ㅋㅋ

제 미흡한 프로그래밍 경험을 되새겨 보면, 추상화를 자주 한 부분은 다음과 같습니다. 
1. 추상화가 완료된 로직을 인터페이스 또는 추상클래스로 옮길 경우
    + SOLID 원칙의 리스코프 치환 & Facade Pattern에서도 자주 반복되는 만큼 중요해 보입니다.
2. 추상화를 하는 과정에서 공통되는 로직은 아니지만 존재 해야 하는 경우.
