### DIP
- 의존성 역전 원칙
- 오직 추상화 된 클래스에만 의존해라.
- 의존성 주입을(DI) 통해 의존 관계를 맺는다.

다음은 DIP에 있는 객체들의 관계이다.
```text
                               ↙ Galaxy(class - 구현 객체)
    Owner(class) ➡ Phone(InterFace) 
                               ↖ Iphone(class - 구현 객체)
```
장점


1. Owner는 Phone이 기능을 어떻게 구현했는지 모르고 사용할 수 있다.


2. 구현체가 바뀌어도, 수행 과정이 바뀔 뿐 이지 수행하는 기능은 같기 때문에 Owner에게 영향을 끼치지 못한다.

----

내 궁금증이었던 것들

1. Phone 인터페이스에 기능(메서드)의 추가
   - Owner 클래스에서는 사용하든 안 하든 상관 없다.
   - 구현체에서는 당연하게 그 기능을 구현해 놓아야 한다.


2. Phone을 사용하지 않고 직접적으로 의존성을 가질 경우
    - 구현체가 바뀔 때마다 Owner에서의 의존성을 직접 바꾸어 주어야 한다.    
    `Galaxy galaxy;` ➡️ `Iphone iphone;`
